# Phase 1 — UI Design Contract

> Visual and interaction contract for the Phase 1 minimal sensor dashboard.
> Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Scope:** This UI contract covers only the Phase 1 delivery: live sensor readings per
> zone with freshness timestamps, stale-data flagging, node online/offline status and
> last heartbeat, stuck-sensor indicators, and the system health panel. The full dashboard
> (alerts bar, actuator control, recommendation queue) is Phase 2 scope and must not be
> designed into this phase's components.

---

## Design System

| Property | Value | Source |
|----------|-------|--------|
| Tool | none (manual tokens, no component library) | shadcn gate: not applicable — SvelteKit/Svelte 5, not React/Next.js/Vite |
| Preset | not applicable | — |
| Component library | none — hand-authored Svelte 5 components | objective context |
| Icon library | Lucide (lucide-svelte) — MIT, tree-shakeable, IoT-appropriate glyphs | default recommendation |
| Font | Inter (Google Fonts, self-hosted via `@fontsource/inter`) — neutral, legible on mobile at small sizes | default recommendation |

**Rationale for no component library:** Phase 1 delivers 1 screen with ~4 component
types (zone card, sensor value, status badge, system health row). A full component
library would add bundle weight and coupling with no Phase 1 payoff. Phase 2 can
introduce a component library once the component surface is larger.

## Spacing Scale

Declared values (multiples of 4 only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Icon-to-label gap, inline badge padding |
| sm | 8px | Sensor value row internal spacing, chip padding |
| md | 16px | Zone card internal padding, default element spacing |
| lg | 24px | Between zone cards in the grid, section heading margin |
| xl | 32px | Page horizontal padding (desktop), section breaks |
| 2xl | 48px | Major section breaks (zone grid to system health panel) |
| 3xl | 64px | Not used in Phase 1 — reserved for Phase 2 page-level spacing |

Exceptions:
- Touch targets for status badges and interactive indicators: minimum 44px hit area
  (iOS/Android tap target requirement, WCAG 2.5.5). Visual size may be smaller; padding
  expands the tap zone.
- Bottom safe area inset on mobile PWA: 16px minimum above the home indicator bar
  (env(safe-area-inset-bottom) CSS variable, applied to page bottom padding).

## Typography

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Display | 28px | 600 (semibold) | 1.2 | Zone name heading on zone card |
| Heading | 20px | 600 (semibold) | 1.2 | Section headings ("System Health", "Garden Zones") |
| Body | 16px | 400 (regular) | 1.5 | Sensor labels, status text, timestamps |
| Label | 14px | 400 (regular) | 1.4 | Secondary labels, badge text, "last seen" timestamps |

**Sensor value numerics:** 28px, weight 600, tabular-nums font-variant (prevents layout
shift as values update via WebSocket). Use `font-variant-numeric: tabular-nums` on all
sensor reading elements.

**Stale timestamp copy:** 14px, weight 400, rendered in the semantic stale color (see
Color section). Prepend a warning icon (Lucide `AlertTriangle`, 14px).

**Constraints:**
- Exactly 2 weights are permitted: 400 and 600. No 500/700 allowed.
- Minimum font size anywhere on screen: 14px (label role). Nothing smaller.
- Do not use italic for any status or sensor state — reserve italic for zero-value states
  only (e.g., "--" placeholder when no reading has arrived yet).

## Color

Dark theme. All surfaces are dark; text is light. No light-mode in Phase 1.

| Role | Hex | HSL | Usage |
|------|-----|-----|-------|
| Dominant (60%) | `#0f1117` | hsl(228, 9%, 8%) | Page background, zone card background |
| Secondary (30%) | `#1c1f2b` | hsl(228, 18%, 14%) | System health panel background, card inner surfaces, table rows |
| Accent (10%) | `#4ade80` | hsl(142, 69%, 58%) | See accent reserved-for list below |
| Stale | `#f59e0b` | hsl(38, 92%, 50%) | Stale readings, freshness warning, staleness icon |
| Stuck | `#f97316` | hsl(24, 95%, 53%) | Stuck-sensor indicator, stuck badge background |
| Offline | `#ef4444` | hsl(0, 84%, 59%) | Node offline badge, offline status indicator |
| Online | `#4ade80` | — | Same as Accent — node online indicator dot |
| Muted text | `#6b7280` | hsl(220, 9%, 46%) | Placeholder text, "--" empty values, secondary metadata |
| Surface border | `#2d3149` | hsl(228, 24%, 24%) | Zone card border, dividers |
| Text primary | `#f1f5f9` | hsl(210, 40%, 95%) | All primary readable text |
| Text secondary | `#94a3b8` | hsl(215, 19%, 65%) | Sensor labels, timestamps, secondary copy |

**Color palette rationale:** Dark green accent (`#4ade80`) reads as "farm/nature healthy"
and provides strong contrast against dark surfaces. Amber for stale (warning, not error).
Orange for stuck (more urgent than stale, less than offline). Red for offline (highest
severity in Phase 1).

**Accent reserved for:**
1. Node online status indicator dot (filled circle, 8px)
2. GOOD quality flag badge background (with dark text)
3. Freshness indicator when data is current (under 60 seconds old)
4. No other elements. Do not use accent for interactive buttons, links, or decorative
   elements in Phase 1.

**Destructive:** `#ef4444` (same as Offline) — used only for node offline state. There
are no destructive user actions (no delete, no confirm-to-proceed flows) in Phase 1.

**Quality flag color mapping:**

| Flag | Color | Hex |
|------|-------|-----|
| GOOD | Accent green | `#4ade80` |
| SUSPECT | Amber | `#f59e0b` |
| BAD | Red | `#ef4444` |
| No data | Muted | `#6b7280` |

## Component Inventory

Four component types required in Phase 1. No others.

### 1. ZoneCard

Displays one garden zone's live sensor state.

**Layout:** Full-width on mobile (single column). Two-column grid on screens >= 640px.
Card is 100% width of its grid column. Internal padding: 16px (md).

**Contents (top to bottom):**
1. Zone name — Display role (28px/600), truncated with ellipsis if > 20 chars
2. Freshness timestamp — "Updated 23s ago" or "STALE — last updated 7m ago" — Label role
   (14px/400). Normal state: text-secondary color. Stale state: stale color + AlertTriangle
   icon (14px).
3. Three sensor rows (one per sensor type):
   - Soil Moisture: `droplets` icon (16px) + label "Moisture" + value "42.3%" + quality
     flag badge
   - pH: `flask-conical` icon (16px) + label "pH" + value "6.8" + quality flag badge
   - Temperature: `thermometer` icon (16px) + label "Temp" + value "22°C" + quality flag
     badge
4. Stuck-sensor indicator (conditional): only visible if any sensor on this zone is stuck.
   Amber/orange row at card bottom: `alert-triangle` icon + "Stuck sensor detected" in
   14px/400.

**Stale state:** When any reading is older than 5 minutes, the entire card gets a 1px
border in the stale color (`#f59e0b`). The freshness timestamp switches to stale copy and
stale color. Sensor values remain visible but are visually dimmed (opacity: 0.5) to signal
unreliability.

**No-data state:** When a zone has never received a reading (just configured), all sensor
value slots show "--" in muted text color (`#6b7280`), and the freshness timestamp shows
"No data received". No stale border is shown (stale only applies when data existed but
expired).

**Quality flag badge:** Inline pill badge, 12px/400, positioned inline after the sensor
value. 4px horizontal padding (xs), 2px vertical padding. Background color per quality
flag mapping. Dark text (`#0f1117`) on GOOD/SUSPECT; light text (`#f1f5f9`) on BAD.

### 2. SensorValue

Reusable sub-component used inside ZoneCard for each sensor row.

**Layout:** Horizontal flex row, space-between. Left: icon + label (text-secondary, 14px).
Right: numeric value + quality badge.

**Real-time update behavior:** Value updates via WebSocket without page refresh. No flash
animation on update. Numeric value uses `tabular-nums` to prevent layout shift. Value
transition: none (instant update — farm context, not financial dashboard).

### 3. NodeHealthRow

Displays one edge node's health state in the System Health panel.

**Layout:** Full-width row. Horizontal flex: [status dot] [node name] [spacer] [last
heartbeat] [status badge]

**Status dot:** 8px filled circle. Color: accent green (online), red (offline). No
animation.

**Status badge:** Pill badge, 12px/400. "ONLINE" in accent green. "OFFLINE" in red with
inverted background.

**Last heartbeat:** Label role, text-secondary. Format: "Heartbeat 47s ago" for recent.
"Last seen 14m ago" for stale. "Never" for nodes that have not yet reported.

**Offline threshold:** Node is shown as OFFLINE when 3 consecutive 60-second heartbeats
are missed (180 seconds without heartbeat). This is a display state driven by the backend;
the UI simply renders the state it receives via WebSocket.

**Missing heartbeat approaching threshold:** No intermediate warning state in Phase 1.
Online/Offline only. (Phase 2 may add a "degraded" state.)

### 4. SystemHealthPanel

Container section for all NodeHealthRow components.

**Layout:** Full-width section below the zone grid. Background: secondary color
(`#1c1f2b`). Padding: 16px (md). Section heading: "System Health" in Heading role
(20px/600), text-primary color.

**Empty state:** If no nodes have ever reported, show: heading + "No nodes connected" in
body role, muted text. No error state — this is expected during initial setup.

## State Definitions

Phase 1 has exactly five UI states. Every sensor reading and every node is always in one
of these states. No other states exist in Phase 1.

| State | Trigger condition | Visual treatment |
|-------|------------------|-----------------|
| Current | Reading age < 5 minutes AND quality GOOD | Normal display, accent freshness indicator |
| Stale | Reading age >= 5 minutes | Amber border on card, dimmed values (opacity 0.5), stale copy |
| Suspect | Quality flag = SUSPECT | SUSPECT badge (amber) inline, no card border change |
| Bad | Quality flag = BAD | BAD badge (red) inline, no card border change |
| Stuck | Same value for 30+ consecutive readings | Stuck indicator row at card bottom (orange), STUCK badge |
| No data | Zone configured but no readings received | "--" placeholders, muted text, no stale border |
| Offline | Node missed 3 × 60s heartbeats | NodeHealthRow: red dot, OFFLINE badge, "Last seen N ago" |
| Online | Node heartbeat within last 60s | NodeHealthRow: green dot, ONLINE badge |

Note: Stale and Stuck can be simultaneously true (a stuck sensor with old data). Both
indicators are shown when both conditions are met.

## Real-Time Behavior Contract

The dashboard receives sensor updates and heartbeats via a single WebSocket connection.

**Connection lifecycle:**
- On page load: establish WebSocket to `/ws/dashboard` (path TBD at implementation).
- Indicator dot in header reflects WebSocket state: green (open), amber (connecting /
  reconnecting), red (closed > 5 seconds).
- On disconnect: sensor values remain visible at last known state. Stale logic continues
  to run client-side based on `received_at` timestamp stored on each reading. Do NOT clear
  values on disconnect — stale display is more useful than blank.
- Reconnect: automatic exponential backoff. On reconnect, the server sends a full state
  snapshot before streaming deltas.

**Update frequency (from requirements):** Nodes poll sensors on a configurable interval.
Assume 60-second poll interval as default. Dashboard should not flash or re-render the
full card on each update — only the changed sensor value field updates.

**Svelte 5 reactivity:** Use runes (`$state`, `$derived`) for per-zone sensor state.
A `Map<zoneId, ZoneState>` reactive store is the recommended structure. Each ZoneCard
receives its zone's state as a prop and derives display states (stale, stuck) from
`$derived`.

## PWA Constraints

Phase 1 delivers the HTTPS endpoint (INFRA-09) required for PWA install. The service
worker and offline shell are Phase 2 scope (UI-05). However, the Phase 1 dashboard must
not block or conflict with Phase 2 PWA registration.

**Phase 1 PWA requirements (minimum):**
- `manifest.webmanifest` present with: `name: "Farm Dashboard"`, `short_name: "Farm"`,
  `display: "standalone"`, `background_color: "#0f1117"`, `theme_color: "#1c1f2b"`,
  `icons` array with at least 192px and 512px maskable icons.
- `theme-color` meta tag in `<head>`: `#1c1f2b` (matches header; prevents system chrome
  color clash on Android).
- `viewport` meta: `width=device-width, initial-scale=1, viewport-fit=cover` (the
  `viewport-fit=cover` is required for iOS safe area inset CSS variables to work).
- No service worker in Phase 1. Phase 2 will register one.

**"Add to Home Screen" on iOS:** Requires HTTPS (Caddy, INFRA-09) and the manifest.
Without a service worker, the PWA will not work offline in Phase 1 — this is acceptable
and expected. The Phase 1 goal is only to make the URL installable.

## Accessibility Minimum

Phase 1 is a single-user operational tool, but these minimums are non-negotiable:

- All color-coded states (stale, stuck, offline, quality flags) must have a text or icon
  companion. Never convey state by color alone.
- ARIA live regions: the zone sensor values section should be `aria-live="polite"` so
  screen readers announce updates without interrupting. The WebSocket connection status
  dot needs `role="status"` and `aria-label` per the copywriting contract above.
- Minimum contrast: 4.5:1 for body/label text against their backgrounds. The chosen
  palette meets this: `#f1f5f9` on `#0f1117` = 15.9:1; `#94a3b8` on `#0f1117` = 7.4:1;
  `#4ade80` on `#0f1117` = 8.5:1.
- Touch targets: 44px minimum (see Spacing Exceptions above).
- Semantic HTML: use `<main>`, `<section>`, `<h1>`/`<h2>` heading hierarchy. Each zone
  card is a `<section>` with `aria-label="{zone name}"`.
